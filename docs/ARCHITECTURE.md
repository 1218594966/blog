# 项目架构

## 当前定位

这个仓库现在承担三层职责：

1. 主站
   负责个人介绍、博客、AI 对话区、联系入口和统一品牌表达。
2. 内容后台
   负责更新站点文案、项目、文章、社交链接和 AI 配置。
3. 扩展入口
   通过 `/tools`、`/projects`、`/lab` 为后续的小工具和独立项目提供统一入口。

## 目录说明

```text
.
├─ content/                 # 仓库内默认内容模板
├─ deploy/                  # 部署、更新、自动同步脚本
├─ docs/                    # 公开文档、路线图、架构说明
├─ public/                  # 前台页面、后台页面、静态资源
│  ├─ assets/               # 样式和前端脚本
│  ├─ tools/                # 工具中心入口
│  ├─ projects/             # 项目中心入口
│  └─ lab/                  # 实验区入口
├─ storage/                 # 运行时数据，已被 git 忽略
├─ server.js                # Express 服务入口
└─ ecosystem.config.cjs     # PM2 运行配置
```

## 路由分层建议

- `/`
  主站首页，承担品牌表达和内容导航。
- `/blog/...`
  后续可扩展为独立文章详情页。
- `/tools/...`
  轻量功能、小工具、效率插件。
- `/projects/...`
  完整应用、长期维护项目、专题站。
- `/lab/...`
  原型、实验、动效试验、临时想法。
- `/admin`
  内容后台。

## 内容与运行时数据

- `content/`
  仓库里的默认模板，适合作为初始内容和公开示例。
- `storage/`
  服务器运行后真正写入的内容、留言和 AI 私密配置。

这样做的好处是：

- GitHub 里的公开仓库保持干净。
- 服务器上的后台修改不会被 `git pull` 覆盖。
- AI Key 等敏感信息不会进入版本库。

## 后续推荐演进

如果后面功能继续增长，建议按下面顺序演进，而不是一次性大改：

1. 把 `server.js` 拆成 `server/` 目录。
2. 把 `public/assets/site.js` 拆成内容渲染、交互、AI 三部分。
3. 把 `public/assets/admin.js` 拆成表单状态、重复项编辑器、API 通信三部分。
4. 当 `/tools` 和 `/projects` 下子页面明显增多时，再考虑升级为更正式的多应用结构。
